Documentation

This repo is a Cursor rules library and install toolkit. It ships 296 technology-specific rules, 24 personal workflow rules, and 152 preset stacks — so you can give Cursor consistent, stack-aware AI guidance in any project.

296 library rules

Best practices for React, ClickHouse, Kafka, FastAPI, AWS, and more. Generated with Exa + LLM.

24 personal rules

Hand-written team conventions: architecture, RAG, security, distributed systems.

152 preset stacks

One-command bundles for common project types — data eng, gen-ai, mobile, backend.

You do not need to clone this repo. Install rules directly from GitHub with curl, the CLI, or the web catalog.

What are Cursor rules?

Cursor rules are persistent instructions that tell the AI how to behave in your project. They live in your-project/.cursor/rules/ as .mdc files. 

your-project/
  .cursor/
    rules/
      base.mdc       ← always-on team behavior
      react.mdc      ← applies when editing React files
      clickhouse.mdc ← applies when editing SQL/analytics code

Each file has YAML frontmatter + markdown content:

---
description: React best practices
globs: **/*.{jsx,tsx}
alwaysApply: false
---

# React rules
- Prefer functional components
- ...
FieldPurpose
descriptionWhat the rule covers — Cursor uses this to decide relevance
globsFile patterns — rule activates when editing matching files
alwaysApply: trueAlways included in AI context (e.g. your base rule)

Without rules, the AI gives generic answers. With rules, it follows your stack's idioms, team conventions, and domain patterns.

Key concepts

1. Library rules (rules-mdc/)

Technology-specific best practices — one .mdc file per library or framework.

  • Answers: "How should I write code for this technology?"
  • Examples: react.mdc, clickhouse.mdc, apache-kafka.mdc
  • Install as: react, clickhouse, fastapi
  • Indexed in: rules.json (names + tags only, not content)

2. Personal / custom rules (rules-custom/)

Hand-written workflow and architecture rules — not tied to a single library.

  • Answers: "How should the AI behave on my team?"
  • Examples: base.mdc, graph-rag.mdc, authentication.mdc
  • Install as: --custom base or custom:base in stacks
  • Indexed in: custom-rules.json

3. Stacks (stacks.json)

Named bundles — a shopping list of rule names, not rule content.

  • Answers: "What rules should I install for this project type?"
  • Install as: --stack data-stack-kafka
  • Can mix library rules and personal rules in one bundle

Rules vs stacks — what's the difference?

Think of it like a menu vs a combo meal.

RuleStack
What it isContent (AI instructions)A shortcut list (no content)
File type.mdc fileEntry in stacks.json
Install asreact, custom:base--stack react-ts
Can mix types?One at a timeYes — e.g. custom:base + react

Example: installing a stack

This stack definition in stacks.json:

{
  "data-stack-kafka": {
    "description": "Apache Kafka event streaming",
    "rules": ["apache-kafka", "docker", "python"]
  }
}

Running --stack data-stack-kafka copies three separate rule files into your project:

my-app/.cursor/rules/
  apache-kafka.mdc   ← from rules-mdc/
  docker.mdc         ← from rules-mdc/
  python.mdc         ← from rules-mdc/
Analogy: Rules are ingredients. Stacks are recipes that say which ingredients to grab.

What each JSON file does

JSON files are indexes only — they describe what exists, not the rule text.

FileRole
rules.jsonMaster list of library rules + tags for search/filter
custom-rules.jsonIndex of personal rules (base, graph-rag, security, …)
stacks.jsonNamed bundles mixing library + personal rules
repo.jsonFork URL config for curl/CLI installers

End-to-end flow

rules.json entry ↓ generate_mdc_files.py (+ Exa + OpenAI) ↓ rules-mdc/{name}.mdc ← actual rule content ↓ install_rules.py / install.sh ↓ your-project/.cursor/rules/ ← Cursor reads these ↓ AI follows your stack + team conventions

As a user, you only touch install commands. Generator files in src/ are for maintainers.

How to install

Curl (no clone, no Python)

# Single library rule
curl -fsSL .../install.sh | bash -s -- react --here

# Preset stack
curl -fsSL .../install.sh | bash -s -- --stack data-stack-kafka --here

# Personal rules
curl -fsSL .../install.sh | bash -s -- --custom base graph-rag --here

# All personal rules
curl -fsSL .../install.sh | bash -s -- --custom-all --here

Python CLI (from clone or GitHub source)

uv run src/install_rules.py list
uv run src/install_rules.py search clickhouse
uv run src/install_rules.py stacks
uv run src/install_rules.py install react fastapi --here
uv run src/install_rules.py install --stack gen-ai-graph-rag --here
uv run src/install_rules.py install --custom base --here --source github

Web catalog

Open Browse Catalog, search rules/stacks, click "Copy curl" or "Copy uv".

All methods copy .mdc files into .cursor/rules/ in your project. Cursor picks them up automatically.

Simple mental model

rules.json → "what library rules exist" custom-rules.json → "what personal rules exist" stacks.json → "recommended combos for project types" rules-mdc/ → the actual library rule files rules-custom/ → the actual personal rule files install → copies .mdc files into your project generate → creates new .mdc files (maintainers only)
In one sentence: JSON files describe what rules exist and how to bundle them; .mdc files are the actual AI instructions; install tooling copies the right ones into your project; Cursor reads them when you code.

For maintainers

To add a new library rule:

  1. Add entry to rules.json
  2. uv run src/generate_mdc_files.py --library new-lib (needs EXA + OpenAI keys in .env)
  3. uv run src/generate_catalog.py
  4. Commit and push — GitHub Pages redeploys automatically

See GUIDE.md for full CLI reference, fork config, and troubleshooting.
